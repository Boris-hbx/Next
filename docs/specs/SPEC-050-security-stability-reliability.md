# SPEC-050: 安全·稳定·可靠 — 研讨结论与行动方案
> 起草日期: 2026-02-26
> 状态: 草稿

## 研讨背景

Next 应用从个人工具走向多用户推广，需要系统性地审视安全、稳定、可靠三个维度。本次研讨目标：**先讨论"做什么、做成什么样"，不讨论"怎么做"。**

研讨团队组成：
- **竞品分析组**：调研 Todoist、TickTick、Notion、Things 3、Microsoft To Do
- **现状审计组**：深入审查 Next 代码和配置
- **最佳实践组**：调研 NIST、OWASP、行业标准

## 一、竞品洞察

### 1.1 认证与账户安全

| 能力 | Todoist | TickTick | Notion | Things 3 | MS To Do | **Next 现状** |
|------|---------|----------|--------|----------|----------|--------------|
| 密码哈希 | 加盐哈希 | 加盐哈希 | 加盐哈希 | 加盐哈希 | 企业级 | **Argon2id ✓** |
| 2FA/MFA | TOTP | TOTP | TOTP + SAML | 无 | 全套 MFA | **无 ✗** |
| OAuth 登录 | Google/Apple | Google/Apple | Google/Apple/SAML | 无 | Microsoft | **无 ✗** |
| 密码找回 | 邮箱重置 | 邮箱重置 | 邮箱/SSO | 邮箱重置 | 多渠道 | **无 ✗** |
| 登录防暴力 | 有 | 有 | 有 | 无 | 有 | **IP+用户双重限流 ✓** |

**结论**：Next 的密码安全做得好（Argon2id + 双重限流），但缺少密码找回和 2FA 这两个用户扩张的前置条件。

### 1.2 数据加密

| 能力 | 竞品标准 | **Next 现状** |
|------|---------|--------------|
| 传输加密 TLS | 全部 TLS 1.2+ | **Fly.io 强制 HTTPS ✓** |
| 静态加密 | AES-256 普遍 | **无 ✗**（SQLite 明文存储） |
| 端到端加密 | 无竞品实现 | 不需要 |

### 1.3 AI 功能安全（对标 Notion AI）

| 能力 | Notion | **Next 现状** |
|------|--------|--------------|
| AI 使用额度 | 免费 20 次/工作区，付费无限 | **5条/分钟限流，无日/月上限 ✗** |
| Token 预算 | ~4K token/请求 | **2048 token 回复上限 ✓** |
| 提示注入防护 | 多层检测（仍有已知漏洞） | **基础净化 + 安全规则 △** |
| AI 输出自动保存 | 是（已被证明是安全漏洞） | **工具调用直接执行 ✗** |
| 数据发送透明度 | 企业可控制 | **无披露、无用户同意 ✗** |

**Notion 的教训**：研究者已证明可通过 PDF 隐藏文本 + markdown 图片渲染 + 搜索工具实现数据外泄。Next 需要避免这种"致命三角"：不受信任的内容 + 工具调用 + 敏感数据访问。

### 1.4 离线与备份

| 能力 | 竞品最佳实践 | **Next 现状** |
|------|-------------|--------------|
| 离线模式 | Things 3 全离线优先，Todoist 完整离线 CRUD | **无离线 ✗** |
| 自动备份 | Todoist 每日 ZIP，Things 本地 10 天 | **每日 VACUUM 备份 + 部署前备份 △** |
| 异地备份 | 竞品均有云端冗余 | **同一磁盘卷，无异地 ✗** |
| 数据导出 | JSON/CSV 一键导出是标配 | **仅差旅 XLSX，无全量导出 ✗** |
| 账户删除 | GDPR 要求，TickTick 7 天内删除 | **无账户删除功能 ✗** |

## 二、现状审计摘要

### 2.1 发现的高风险问题

| # | 问题 | 风险 | 影响 |
|---|------|------|------|
| **H1** | `serve_photo` 路径穿越漏洞 | **高** | 任何登录用户可通过 `../../next.db` 下载整个数据库（含所有用户密码哈希和数据） |
| **H2** | 所有备份在同一磁盘卷 | **高** | 磁盘故障 = 数据全丢（主库 + 备份） |
| **H3** | 开放注册无验证 | **中高** | 攻击者可无限创建账户，消耗资源 |

### 2.2 中风险问题

| # | 问题 | 领域 | 影响 |
|---|------|------|------|
| M1 | CSP 使用 `unsafe-inline` | 安全 | XSS 防护被削弱 |
| M2 | 阿宝无日/月 Token 预算 | 安全 | 恶意用户可烧钱 7200条/天 |
| M3 | 61 处 `.unwrap()` + `panic=abort` | 稳定 | 一次 DB 错误崩溃整个服务 |
| M4 | 单 DB 连接 + Mutex 序列化 | 稳定 | 并发请求阻塞，延迟飙升 |
| M5 | 无 SIGTERM 优雅关闭 | 稳定 | 部署时在途请求丢失 |
| M6 | 大照片 ZIP 导出全在内存 | 稳定 | 256MB VM 可能 OOM |
| M7 | 健康检查不验证数据库 | 可靠 | DB 坏了仍收流量 |
| M8 | 无账户删除 | 隐私 | 潜在法规风险 |
| M9 | 用户数据发送至 Claude API 无披露 | 隐私 | 财务、任务、小票照片均上传 |
| M10 | 限流状态存内存，重启即清零 | 安全 | 攻击者可等部署后重试 |

### 2.3 做得好的地方（继续保持）

- Argon2id 密码哈希 + 时序攻击防护
- HttpOnly + Secure + SameSite=Lax Cookie
- IP 限流 + 用户锁定双层防护
- 全面的安全响应头（HSTS、X-Frame-Options、Referrer-Policy）
- SQL 参数化查询（极少数例外）
- 容器多阶段构建 + 非 root 运行
- WAL 模式 + 外键约束
- 每日自动备份 + 保留 30 天 + 部署前备份

## 三、愿景：Next 要做成什么样

### 3.1 安全愿景

> **让用户放心把最私密的计划和财务数据交给 Next。**

目标状态：
- 零已知高风险漏洞
- 用户数据全链路保护：存储加固、传输加密、AI 调用最小化
- 登录安全达到行业标准：密码找回、可选 2FA
- AI 功能有明确的额度控制和用户知情同意
- 注册有门槛，防止资源滥用

### 3.2 稳定愿景

> **7×24 可用，一个用户的异常不影响其他人。**

目标状态：
- 关键路径无 panic（错误返回友好提示而不是崩溃）
- 数据库并发能力从"单线程"提升到"连接池"
- 外部依赖（Claude API）故障不拖垮核心功能
- 部署零中断

### 3.3 可靠愿景

> **数据不丢、能导出、能恢复。**

目标状态：
- 异地备份，磁盘故障不丢数据（RPO ≤ 1 小时，RTO ≤ 30 分钟）
- 用户可一键导出全部数据
- 用户可删除账户和所有数据
- 备份自动验证，不是"以为备了但其实是坏的"

## 四、行动清单

### 第一优先级：立即要做（安全红线）

这些是"不做会出事"的问题：

| # | 做什么 | 做成什么样 | 领域 |
|---|--------|-----------|------|
| **P1** | 修复 `serve_photo` 路径穿越 | 文件名不允许包含 `..`、`/`、`\`，验证不通过返回 400 | 安全 |
| **P2** | 阿宝 AI 每日额度 | 每用户每日 Token 上限（如 10 万），超限友好提示"明天再来" | 安全 |
| **P3** | 阿宝 API 调用超时 | 单次 Claude API 请求 30 秒超时，避免挂起 | 安全+稳定 |
| **P4** | 对话/消息数量上限 | 每用户最多 100 个对话、每对话最多 200 条消息 | 安全 |
| **P5** | 限流状态持久化 | 登录限流数据存 DB，重启不清零 | 安全 |
| **P6** | 注册增加邀请码机制 | 未持有邀请码不能注册，管理员可生成邀请码 | 安全 |

### 第二优先级：近期要做（基础设施加固）

这些是"扩张前必须到位"的能力：

| # | 做什么 | 做成什么样 | 领域 |
|---|--------|-----------|------|
| **P7** | 异地备份 | 通过 Litestream 持续复制到 S3/R2，RPO ≤ 1 小时 | 可靠 |
| **P8** | 健康检查包含 DB 验证 | `/health` 执行 `SELECT 1` 确认 DB 可用 | 可靠 |
| **P9** | 全量数据导出 | `GET /api/me/export` 返回 JSON（所有待办/记账/对话/联系人等） | 可靠+隐私 |
| **P10** | 账户删除 | `DELETE /api/me` 彻底删除用户所有数据 | 隐私 |
| **P11** | 隐私政策页面 | 明确告知：收集什么数据、存在哪、谁能看、AI 处理说明 | 隐私 |
| **P12** | AI 数据使用同意 | 首次使用阿宝前弹窗告知数据将发送至 Anthropic | 隐私 |
| **P13** | 密码找回机制 | 注册时生成 8-10 个一次性恢复码，用户保存后可用于重置密码 | 安全 |
| **P14** | route handler 消除 `.unwrap()` | 关键路径（注册、登录、创建数据）全部用 Result 返回错误而非 panic | 稳定 |
| **P15** | SIGTERM 优雅关闭 | 收到信号后停止接新连接、等待在途请求完成（最多 30 秒）、关闭 | 稳定 |
| **P16** | SQLite 加固 PRAGMA | `synchronous=FULL` + `secure_delete=ON` + `temp_store=MEMORY` | 安全+可靠 |
| **P17** | 密码策略更新 | 遵循 NIST 800-63B：最低 12 字符、去除大小写数字强制要求、增加常见泄露密码检测 | 安全 |
| **P18** | 外部监控告警 | UptimeRobot / BetterStack 监控，宕机 2 分钟内 Telegram/短信告警 | 可靠 |
| **P19** | 部署零中断 | fly.toml 增加 `[deploy] strategy = "bluegreen"` | 稳定 |

## 五、机会点分析

### 5.1 差异化竞争优势

Next 相比竞品有几个独特机会：

1. **AI 原生的任务管理** — Todoist/TickTick 的 AI 是后加的，Next 的阿宝是核心功能。做好 AI 安全（额度、透明度、用户控制权），就是竞争壁垒。

2. **数据主权** — 竞品全部是 SaaS，用户数据在别人的服务器上。Next 技术上可以做到"数据在你自己的服务器"（自部署能力），这对隐私敏感用户有巨大吸引力。

3. **小而美的信任感** — 独立开发者应用天然有一种"为你而建"的亲切感。把隐私政策写得像人话、让数据导出一键完成、账户删除干净彻底，这些"小事"会建立深厚信任。

### 5.2 推广前的安全底线

在向更多人推广之前，以下是**不可妥协的底线**：

- [ ] 路径穿越漏洞修复（P1）
- [ ] AI 额度控制到位（P2/P3/P4）
- [ ] 注册有门槛（P6）
- [ ] 数据可导出（P9）
- [ ] 账户可删除（P10）
- [ ] 隐私政策公开透明（P11）
- [ ] 异地备份（P7）

## 六、遗留问题

以下问题本次研讨不展开，记录待后续处理：

1. **阿宝时区问题** — 当前基于服务器时区（北京），不跟随用户所在时区。需要前端传时区给后端。
2. **CSP `unsafe-inline` 清理** — 需要重构前端 JS/CSS 以支持 nonce-based CSP，工作量大，适合作为独立 spec 处理。
