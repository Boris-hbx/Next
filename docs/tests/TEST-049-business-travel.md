# 测试方案: 差旅管理 (SPEC-049)

> 创建日期: 2026-02-24
> 关联规格: `docs/specs/SPEC-049-business-travel.md`
> 测试环境: staging (https://next-boris-staging.fly.dev)

## 测试策略

本测试方案分 7 个维度，共 80+ 个测试点。每个维度对应 spec 中的一个核心能力。
开发完成后按顺序执行，全部通过才可合入 main 部署 production。

**测试账号准备：**
- 账号 A: 主账号（差旅创建者）
- 账号 B: 助理账号（协作者），与 A 互为好友
- 账号 C: 无关账号（非好友），用于权限边界测试

## 十一、端到端场景（完整流程）

模拟真实使用场景的完整操作链路。

### E2E-01: 完整差旅生命周期

```
1. 创建差旅"回国述职"，2/29-3/5，目的地北京，币种 CAD
2. 添加 6 个条目:
   - 2/29 ✈️ AC025 温哥华→北京 ¥6,800
   - 3/1  🏨 北京希尔顿 4晚 ¥3,560
   - 3/1  🚕 机场→酒店 ¥85
   - 3/2  📋 述职汇报 (无金额)
   - 3/2  🍽️ 团队聚餐 ¥680 (无需报销)
   - 3/5  ✈️ AC026 北京→温哥华 ¥7,200
3. 给机票条目上传行程单照片
4. 给酒店条目上传发票照片
5. 核对报销汇总: 总计¥18,325, 无需报销¥680
6. 把两张机票改为"已提交"
7. 核对汇总变化: 已提交¥14,000
8. 导出 CSV → 打开验证
9. 下载票据 zip → 解压验证照片
10. 分享给助理(editor)
11. 助理登录 → 看到行程 → 把机票改为"已批准"
12. 回到自己账号 → 看到状态已更新
```

预期: 全流程无报错，数据一致，导出内容正确。

### E2E-02: 助理报销工作流

```
1. 账号A 创建差旅并添加 10 个条目
2. A 分享给 B(editor)
3. B 登录 → 看到差旅
4. B 逐条查看票据 → 把有发票的改为"已提交"
5. B 导出 CSV 给财务
6. A 登录 → 确认所有状态变更正确
7. A 把已报销的改为"已批准"
8. 最终报销汇总：全部绿色
```

### E2E-03: 多差旅管理

```
1. 创建 3 个差旅: 已完成、进行中、未来
2. 列表排序正确
3. 每个差旅添加若干条目
4. 返回列表 → 每张卡片金额和进度条分别正确
5. 删除已完成的差旅 → 列表只剩 2 个
```
